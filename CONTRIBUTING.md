# 参与共建

这个 Skill 的目标是成为国内自媒体发布前自审的开放标准。规则的准确性靠真实反馈打磨，欢迎三种参与方式：

## 1. 报告误报 / 漏报（最有价值）

开 issue 说清三件事即可，不需要懂技术：

- 内容片段（可脱敏）；
- 审核结论说了什么；
- 实际发生了什么（平台放行了 / 被处罚了，附通知截图更好）。

## 2. 提需求

想要新平台（快手/B站/公众号）、新行业规则、新功能，直接开 issue 用一句话描述场景。ROADMAP 的优先级按真实需求排序。

## 3. 贡献规则

规则库对来源有硬要求，贡献时请附上：

- **规则描述**：什么情况违规、什么情况不算；
- **官方依据**：来源文件名称 + 原文引用 + 链接（平台规则中心、社区公约、法规原文）。没有官方来源的"经验之谈"也欢迎，但会标注为社区经验层，不与官方规则混淆；
- **正反例子**：至少一个应命中、一个不应命中的例子。

规则数据在上游维护仓库统一管理（来源登记 → 原子规范 → 编译规则 → 回归测试），本仓库的 `references/rules-*.md` 和 `scripts/terms.json` 由上游生成。直接对这些文件提 PR 也可以，维护者会同步回上游。

## 授权说明

提交贡献（PR、issue 中的规则、案例、词表等内容）即表示你同意：你的贡献按本项目对应部分的协议发布——代码为 MIT，文档与规则内容为 CC BY-NC 4.0（含项目声明的附加许可），并允许项目维护者在项目授权场景中使用你的贡献。

## 行为底线

不接受任何绕审技巧的贡献（谐音表、拆字方案、暗号词典）。这个项目帮创作者合规表达，不帮任何人欺骗审核。
